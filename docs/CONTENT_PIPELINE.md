# Content pipeline

Этот документ описывает путь любого объекта Cogwild от идеи до проверенного игрового контента.

## 1. Идея

Объект создаётся в Notion и получает:

- русское название;
- технический ID в `snake_case`;
- категорию;
- краткое описание;
- требования к модели и анимации;
- текущий статус.

## 2. Техническая карточка

До начала моделирования фиксируются:

- тип: item, block, entity или armor;
- размер текстуры;
- коллизия и габариты;
- направления поворота;
- состояния и анимации;
- инвентарь, энергия и жидкости, если они нужны;
- связанные рецепты и достижения.

## 3. Модель и текстура

Исходник Blockbench хранится в `art_source/blockbench/`.

Экспортируемые файлы размещаются в `src/main/resources/assets/cogwild/`:

- GeckoLib geometry: `geo/`;
- GeckoLib animations: `animations/`;
- textures: `textures/`;
- обычные item/block models: `models/`;
- block states: `blockstates/`.

## 4. Реализация

Для нового контента добавляются необходимые регистрации и классы. Минимальный набор для обычного блока:

- запись в `ModBlocks`;
- `BlockItem` в `ModItems`;
- blockstate;
- block model;
- item model;
- loot table;
- перевод на русский и английский;
- рецепт, если предусмотрен дизайном.

## 5. Проверка

Перед слиянием Pull Request:

- проект собирается через `./gradlew build`;
- игра запускается;
- объект выдаётся командой `/give`;
- блок корректно ставится и ломается;
- модель и текстура не имеют отсутствующих граней;
- переводы отображаются;
- сервер не загружает клиентские классы;
- в логе нет ошибок Cogwild.

## 6. Статусы

Рекомендуемый поток:

`Идея → Концепт → Моделирование → Текстурирование → Реализация → Тестирование → Готово`

Статус `Готово` ставится только после проверки в игре и успешной CI-сборки.
